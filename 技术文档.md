# 广告商业平台Recall回调处理系统 技术文档

## 1. 系统概述

### 1.1 项目背景
为广告商业平台提供统一的Recall回调处理服务，实现用户回调信息的接收、存储、查询和实时推送功能。

### 1.2 技术选型
- **后端语言**: Go 1.21+
- **Web框架**: Gin
- **ORM框架**: GORM (支持MySQL/PostgreSQL)
- **前端**: HTML + JavaScript (Axios)
- **数据库**: MySQL 8.0+ (可切换PostgreSQL)

## 3. 数据库设计

### 3.1 ER图

```
┌─────────────┐       ┌─────────────┐       ┌─────────────────┐
│   users     │       │   tokens    │       │ recall_records  │
├─────────────┤       ├─────────────┤       ├─────────────────┤
│ id (PK)     │──┐    │ id (PK)     │       │ id (PK)         │
│ username    │  │    │ user_id(FK) │───────│ recall_svc_name │
│ password    │  │    │ token       │       │ platform        │
│ notify_url  │  │    │ expires_at  │       │ user_name       │
│ created_at  │  └───►│ created_at  │       │ params          │
│ updated_at  │       └─────────────┘       │ created_at      │
└─────────────┘                              └─────────────────┘
```

### 3.2 表结构

#### users 用户表
```sql
CREATE TABLE users (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    recall_service_name VARCHAR(64) NOT NULL UNIQUE COMMENT '服务用户名',
    password VARCHAR(255) NOT NULL COMMENT '加密密码',
    notify_url VARCHAR(512) DEFAULT '' COMMENT '通知回调URL',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_recall_service_name (recall_service_name)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### tokens Token表
```sql
CREATE TABLE tokens (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT UNSIGNED NOT NULL,
    token VARCHAR(128) NOT NULL UNIQUE COMMENT 'Token值',
    expires_at DATETIME NOT NULL COMMENT '过期时间',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_token (token),
    INDEX idx_user_id (user_id),
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### recall_records 回调记录表
```sql
CREATE TABLE recall_records (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    recall_service_name VARCHAR(64) NOT NULL COMMENT '服务用户名',
    platform VARCHAR(64) NOT NULL COMMENT '平台来源',
    user_name VARCHAR(128) NOT NULL COMMENT '授权用户名称',
    params TEXT COMMENT '完整参数JSON',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_recall_service_name (recall_service_name),
    INDEX idx_platform (platform),
    INDEX idx_user_name (user_name),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 4.2 Recall回调接口

#### 4.2.1 接收回调
```
GET /recall
GET /recall?state={urlcode_params}&other_params=value

直接访问说明示例:
{
    "code": 0,
    "message": "Recall接口使用说明",
    "data": {
        "description": "回调接口使用说明",
        "format": "/recall?state={urlcode(recall_service_name=XXX&platform=XXX&user_name=XXX)}&other_params",
        "required_params": ["recall_service_name", "platform", "user_name"],
        "optional_params": ["other_params"]
    }
}

正确回调处理 (200):
{
    "code": 0,
    "message": "回调处理成功",
    "data": {
        "record_id": 12345
    }
}

参数缺失 (400):
{
    "code": 2001,
    "message": "缺少必填参数: recall_service_name, platform",
    "data": {
        "missing_params": ["recall_service_name", "platform"]
    }
}

格式错误 (400):
{
    "code": 2002,
    "message": "state参数格式错误，请使用URL编码",
    "data": null
}
```

#### 4.2.2 查询回调记录
```

Authorization: Bearer <token>

Query Parameters:
- recall_service_name: string (可选)
- platform: string (可选)
- user_name: string (可选)
- page: int (可选, 默认1)
- page_size: int (可选, 默认20, 最大100)

Response (成功 - 200):
{
    "code": 0,
    "message": "查询成功",
    "data": {
        "total": 100,
        "page": 1,
        "page_size": 20,

                "id": 1,

                "user_name": "user_123",
                "params": "{\"campaign_id\":\"123\",\"ad_id\":\"456\"}",
                "created_at": "2026-04-28T14:30:00+08:00"
            }
        ]
    }
}

Response (无数据 - 200):
{
    "code": 0,
    "message": "查询成功",
    "data": {
        "total": 0,
        "page": 1,
        "page_size": 20,
```

## 5. 核心业务逻辑

### 5.1 Recall回调处理流程

```
┌──────────────────────────────────────────────────────────────────┐
│                      Recall回调处理流程                           │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 接收请求                                                     │
│      │                                                           │
│      ▼                                                           │
│  2. 检查state参数是否存在                                       │
│      │                                                           │
│      ├─── 不存在 ──► 返回: "缺少state参数"                       │
│      │                                                           │
│      ▼                                                           │
│  3. URL解码state参数                                            │
│      │                                                           │
│      ├─── 解码失败 ──► 返回: "status格式错误，需URL编码"          │
│      │                                                           │
│      ▼                                                           │
│  4. 解析key=value对                                              │
│      │                                                           │
│      ├─── 解析失败 ──► 返回: "status内容格式错误"                 │
│      │                                                           │
│      ▼                                                           │
│  5. 检查必填参数(recall_service_name, platform, user_name)       │
│      │                                                           │
│      ├─── 缺失参数 ──► 返回: "缺少参数: xxx, yyy"                │
│      │                                                           │
│      ▼                                                           │
│  6. 查询用户notify_url                                           │
│      │                                                           │
│      ▼                                                           │
│  7. 存储回调记录到数据库                                          │
│      │                                                           │
│      ▼                                                           │
│  8. 异步推送通知(如果notify_url存在)                               │
│      │                                                           │
│      ▼                                                           │
│  9. 返回成功响应                                                  │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 5.2 Token鉴权流程

```
┌──────────────────────────────────────────────────────────────────┐
│                      Token鉴权流程                                │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 从Header提取Token                                            │
│      Authorization: Bearer <token>                              │
│      │                                                           │
│      ├─── 无Token ──► 返回401: "未提供Token"                     │
│      │                                                           │
│      ▼                                                           │
│  2. 验证Token格式                                                 │
│      │                                                           │
│      ├─── 格式错误 ──► 返回401: "Token格式错误"                  │
│      │                                                           │
│      ▼                                                           │
│  3. 查询Token是否存在且未过期                                     │
│      │                                                           │
│      ├─── 不存在/已过期 ──► 返回401: "Token无效或已过期"         │
│      │                                                           │
│      ▼                                                           │
│  4. 获取用户信息，注入请求上下文                                  │
│      │                                                           │
│      ▼                                                           │
│  5. 继续处理业务逻辑                                              │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 5.3 通知推送流程

```
┌──────────────────────────────────────────────────────────────────┐
│                      异步通知推送流程                             │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 创建回调记录后触发                                            │
│      │                                                           │
│      ▼                                                           │
│  2. 查询用户的notify_url                                         │
│      │                                                           │
│      ├─── 无notify_url ──► 结束                                  │
│      │                                                           │
│      ▼                                                           │
│  3. 构建推送请求                                                  │
│      POST {notify_url}                                           │
│      Content-Type: application/json                             │
│      Body: {                                                     │
│          "recall_service_name": "xxx",                          │
│          "platform": "xxx",                                      │
│          "user_name": "xxx"                                     │
│      }                                                           │
│      │                                                           │
│      ▼                                                           │
│  4. 异步发送HTTP请求                                              │
│      │                                                           │
│      ├─── 超时(5s) ──► 记录日志，继续执行                        │
│      ├─── 失败 ──► 重试3次，间隔2s                               │
│      └─── 成功 ──► 记录日志                                      │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 7. 安全措施

### 7.1 密码安全
- 使用bcrypt加密，cost=10
- 密码最小长度8位，需包含字母和数字

### 7.2 Token安全
- 使用HS256签名算法
- Token有效期24小时
- 刷新Token时使旧Token失效

### 7.3 SQL注入防护
- 使用GORM参数化查询
- 不拼接SQL字符串

### 7.4 HTTP安全
- 通知推送支持HTTPS
- 验证URL格式

## 8. 错误码定义

| 错误码 | 说明 |
|--------|------|
| 0 | 成功 |
| 1001 | 用户名已存在 |
| 1002 | 用户名或密码错误 |
| 1003 | Token无效或已过期 |
| 1004 | 未提供Token |
| 1005 | Token格式错误 |
| 2001 | 缺少必填参数 |
| 2002 | state参数格式错误 |
| 2003 | 参数解析错误 |
| 3001 | 通知URL格式错误 |
| 5001 | 服务器内部错误 |
